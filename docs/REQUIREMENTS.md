## R001 推送与保活
- **场景**：App 关闭/后台后收不到任何消息
- **期望**：关了 App 也能收到通知，有声音提醒
- **方案草案**：
  - 层1：Android 前台服务保 WebSocket 不被杀
  - 层2：FCM 推送（App 被强杀也能收到）
  - 层3：离线消息补拉（重连时补漏）
- **状态**：待确认

## R002 跨实例消息通知
- **场景**：正在跟翎翼聊天，翎绛发来消息，看不到
- **期望**：
  - 消息留在翎绛的对话里，不串到翎翼
  - 底部导航"对话"图标显示未读 badge
  - 实例选择器显示哪些实例有未读消息
  - App 在后台 → 弹系统通知，点击跳到对应实例
  - App 在前台 → 顶部横幅提示，点击切换实例
- **状态**：待确认

## R003 长时间任务执行
- **场景**：在 App 里让翎绛做开发任务（可能跑 15+ 分钟），切后台后断线丢结果
- **期望**：任务在服务端正常跑完，结果能补拉回来
- **依赖**：R001（保活/推送）
- **状态**：待确认

## R004 设置页实例管理重构
- **场景**：设置页的"添加实例"让用户手动填名字+URL，但中继架构下实例是自动注册的，这个功能无用且误导
- **期望**：
  - 去掉手动"添加实例"
  - 设置页只配置：中继服务器地址 + 认证 Token（一次性）
  - 实例列表从中继服务器自动获取，自动显示上下线状态
  - 保留实例的"备注/别名"编辑能力（可选）
- **状态**：待确认

## R005 自动连接
- **场景**：每次打开 App 都要手动点"连接"，设置页还有添加/连接/断开按钮，全是多余操作
- **期望**：
  - 打开 App 直接自动连接中继服务器，零操作
  - 移除设置页的"添加实例"、"连接"、"断开"按钮
  - 中继地址和 Token 内置，不需要用户填写
  - 断线自动重连（已有）
  - 连接状态在状态栏实时显示即可
- **状态**：待确认

## R006 App 接入认证
- **场景**：任何人拿到 APK 就能连上中继服务器，操控 AI 实例，安全风险大
- **期望**：
  - App 首次打开需要输入一个接入码/密码才能使用
  - 认证通过后本地保存，后续自动连接（和 R005 不冲突）
  - 中继服务器验证接入码，拒绝未认证的客户端
  - 可以随时在中继服务器端吊销/更换接入码
- **备选方案**：
  - A：简单密码（一个共享密码，够用）
  - B：邀请码机制（生成一次性邀请码，用完失效）
  - C：设备绑定（首次认证后绑定设备 ID，其他设备用不了）
- **状态**：待确认

## R007 翎云自动运维（中继服务器主动修复）
- **场景**：翎绛/翎翼等实例异常离线，用户不一定能及时处理
- **期望**：
  - 翎云作为中继服务器 + OpenClaw 实例，检测到其他实例离线时主动介入
  - 通过 SSH/Tailscale 连到故障机器，诊断并尝试修复（重启服务、清理磁盘等）
  - 修复结果通知用户（配合 R001 推送）
  - 无法自动修复时，告警通知用户并附上诊断信息
- **前提**：
  - 翎云需要有到其他实例的 SSH 访问权限
  - 定义翎云的自动修复权限边界（能做什么、不能做什么）
- **依赖**：R001（推送通知）
- **状态**：待确认

## R008 对话页实例切换 + 会话上下文
- **场景**：
  1. 对话页不能直接切实例，要跑到实例管理页去切，路径太长
  2. 切了实例后对话是空白的，之前的聊天记录全没了
- **期望**：
  - 对话页顶部有实例选择器（下拉或横滑），一键切换
  - 每个实例的对话记录独立保存，切回来还在
  - 对话记录本地持久化（SQLite/SharedPreferences），App 重启也不丢
  - 切换实例时保留滚动位置
- **状态**：待确认

## R009 实例 API 状态监控
- **场景**：多个实例用不同的 API（yunyi、packy、百炼等），不知道谁的 API 快到期了、余额多少
- **期望**：
  - 每个实例上报自己的 API 信息：提供商名称、到期时间、余额/配额
  - 态势页/实例卡片显示 API 状态（正常/即将到期/已过期）
  - 即将到期（≤3天）时主动告警推送
  - 实例详情页能看到完整的 API 信息列表
- **实现思路**：
  - relay client 的 health_report 新增 api 字段
  - OpenClaw 客户端定期检测 API 有效期（调用各家余额接口）
  - App 端展示 + 告警阈值判断
- **状态**：待确认

## R010 ⭐ 大任务执行反馈（重点）
- **场景**：
  1. 对话中发一个复杂任务，界面卡住，看不到任何过程，不知道是在干活还是挂了
  2. Web 端能看到实时输出（thinking、tool calls、中间结果），App 端完全没有
  3. 超过一定时间（可能是 WebSocket 超时或 OpenClaw 超时），回复直接消失，什么都不返回
- **期望**：
  - **实时过程展示**：像 Web 端一样，能看到 AI 正在思考、正在调用工具、工具返回了什么
  - **流式输出**：回复一边生成一边显示，不要等全部完成才出来
  - **超时保护**：
    - 超时前给用户提示（"任务较大，预计还需 X 分钟"）
    - 超时后不要静默丢失，要明确告知"任务超时"并提供重试选项
    - 可配置超时时间（默认可能太短）
  - **进度指示**：
    - 至少要有动态的"正在处理..."动画，区别于完全卡死
    - 最好能显示当前步骤（"正在读取文件..."、"正在编译..."）
  - **断线恢复**：任务跑到一半 WebSocket 断了，重连后能拿回结果（关联 R001/R003）
- **优先级**：🔴 最高 — 这个不解决，App 对话功能基本不可用
- **状态**：待确认
